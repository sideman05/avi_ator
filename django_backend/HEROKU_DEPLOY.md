# Heroku deployment checklist and H14 / "Couldn't find that process type (web)" fix

This file lists quick checks and commands to make sure the app runs on Heroku and to fix the error:

    Error: Couldn't find that process type (web)

Common cause: Heroku did not see a `Procfile` with a `web` process type in the Git repo you pushed.

Quick verification and fixes

- Confirm the `Procfile` exists in the Git repository you push to Heroku and contains a `web:` line:

```bash
# list tracked files named Procfile
git ls-files | grep -i Procfile || true

# show Procfile content
cat Procfile
```

- If your Django app lives in the `django_backend` subdirectory (as in this repo), you must either:

  1. Put `Procfile` at the repository root (the directory you push to Heroku), commit and push; OR
  2. Push the `django_backend` subdirectory to Heroku using git subtree or set the git remote from inside that subdir.

Examples

# Option A — move `Procfile` to repo root and push (recommended when using single Heroku app from root):

```bash
cp django_backend/Procfile ./
git add Procfile
git commit -m "Procfile at repo root for Heroku"
git push heroku main
```

# Option B — push the `django_backend` subdirectory directly using git subtree:

```bash
# from repository root
git subtree push --prefix django_backend heroku main
```

# Option C — set Heroku remote from inside the subdirectory and push there:

```bash
cd django_backend
heroku git:remote -a <your-heroku-app-name>
git push heroku main
```

After pushing, scale the web dyno:

```bash
heroku ps:scale web=1 --app <your-heroku-app-name>
heroku ps --app <your-heroku-app-name>
```

Debugging runtime failures

- Tail logs to see startup errors and crashes:

```bash
heroku logs --tail --app <your-heroku-app-name>
```

- Check recent releases and release-phase output:

```bash
heroku releases --app <your-heroku-app-name>
```

- Check buildpacks and environment:

```bash
heroku buildpacks --app <your-heroku-app-name>
heroku config --app <your-heroku-app-name>
```

Procfile requirements

- Must be named exactly `Procfile` (no extension) and tracked by Git.
- The file should contain a `web:` process. Example for this project:

```
web: gunicorn aviator_backend.wsgi:application --bind 0.0.0.0:$PORT --workers 3
release: python manage.py migrate --noinput
```

Other notes

- If Heroku build fails at `collectstatic`, examine build output and fix missing static references. You can temporarily set `DISABLE_COLLECTSTATIC=1` to skip during debugging but production should run `collectstatic`.
- If your repo root is different from the Django project root, pushing the correct subtree or moving `Procfile` is the usual fix.
